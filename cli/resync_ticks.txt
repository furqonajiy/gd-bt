# =========================================================================
# RESYNC XAUUSD TICK HISTORY (MT5 -> data/ticks/), GitHub-safe <=95 MiB parts
# Windows-only: requires the MetaTrader5 package + a RUNNING MT5 terminal
# logged into your broker. PowerShell backtick (`) line continuation.
# =========================================================================
#
# What it does:
#   Pulls REAL tick data from MT5 (copy_ticks_range) month-by-month and writes
#   tab-separated CSV(s) under data/ticks/. With --split-mb it then cuts each
#   month into sequential, line-aligned parts of at most that many MiB and
#   deletes the full month file, so every part stays under GitHub's 100 MiB push
#   limit:
#       data/ticks/XAUUSD_TICK_YYYYMM_p1_ELEV8.csv, _p2, _p3, ...
#   (the exact format + naming the tick backtest already reads). Times are stored
#   in CHART time = GMT+3 (the MT5 server clock), same as the M1 archive.
#
#   This is the archive the TICK-aware backtest consumes:
#     - tools/backtest_hybrid.py  (--ticks data/ticks/XAUUSD_TICK_*_ELEV8.csv) and
#       the CLI snapshots' 2026 backtest sections (5 & 6), and
#     - tools/tick_backtest.py and the tick sweeps.
#
# WHY 95 (not 100):
#   GitHub HARD-REJECTS any file >= 100 MiB on push (and warns above 50 MiB).
#   --split-mb 95 leaves margin so a push never bounces; the committed archive's
#   full parts are ~94 MiB. Raise toward 100 only if you like living dangerously.
#
# IMPORTANT — broker tick window is SHORT:
#   MT5 only returns the ticks YOUR BROKER still stores. Tick history is far
#   shorter than M1 -- often just days to a few weeks (some brokers keep no
#   tick history at all). So set --start-date to the recent window you actually
#   want; requesting more just yields whatever is available. The committed
#   archive currently spans 2026-05-05 -> 2026-06-24 (May + June).
#
# IMPORTANT — --split-mb is a FINAL PACKAGING step:
#   it removes the full month file, so a later --merge re-fetches the whole
#   window instead of resuming. Run it once you have the window you want; do not
#   use it for an archive you grow incrementally.
#
# Dates are CHART-time GMT+3; --end-date is EXCLUSIVE.
#
# NOTE: you usually do NOT need to run this by hand. The backtest sections (5 & 6)
# of the strategy snapshots run tools/backtest_hybrid.py with --sync-charts true
# --sync-ticks true, which BEFORE every backtest refreshes BOTH the M1 archive AND
# the tick archive -- the tick refresh is an APPEND (export_ticks --merge,
# re-split at <=95 MiB), exactly what this snapshot does. Run this standalone only
# to (re)build the tick archive outside a backtest, or for a custom date range.
#
# ===== RESYNC TICKS (run from the repo root) =====
python tools/export_ticks.py `
  --symbol XAUUSD `
  --start-date 2026-05-01 `
  --end-date 2026-06-26 `
  --output-dir data/ticks `
  --mt5-server-offset 3 `
  --split-mb 95 `
  --progress

# --symbol to Market Watch exactly. If the broker clock is not GMT+3, set

# ===== CONTINUE FROM THE LATEST STATE (incremental --merge) =====
# Add only the NEW ticks since the last fetch -- no whole-window re-download.
# --merge resumes from the last recorded tick and appends just the newer ones
# (so ticks already archived but since aged out of the broker's tick window are
# preserved -- --overwrite would lose them). Against the COMMITTED split archive,
# --merge AUTO-REASSEMBLES the _pN parts back into the full month file first, then
# resumes; --split-mb 95 then re-packs into GitHub-safe parts. So "continue" is
# just --merge (+ --split-mb to re-pack). Set the dates to the month you are
# extending (same month = grow it forward; a new month = first fetch of it).
python tools/export_ticks.py `
  --symbol XAUUSD `
  --start-date 2026-06-01 `
  --end-date 2026-07-01 `
  --output-dir data/ticks `
  --mt5-server-offset 3 `
  --merge `
  --split-mb 95 `
  --progress

# TIP -- avoid the reassemble round-trip on every update: keep the FULL working
# month file locally (do NOT commit it; it is >100 MiB) and --merge onto it WITHOUT
# --split-mb day-to-day; run --split-mb 95 only right before you commit/push. Then
# `git add` just the _pN parts. Full file = incremental working copy; parts = what
# you commit.
