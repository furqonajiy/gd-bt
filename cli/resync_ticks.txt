# =========================================================================
# RESYNC XAUUSD TICK HISTORY (MT5 -> data/ticks/), GitHub-safe DAY-WINDOW parts
# Windows-only: requires the MetaTrader5 package + a RUNNING MT5 terminal
# logged into your broker. PowerShell backtick (`) line continuation.
# =========================================================================
#
# What it does:
#   Pulls REAL tick data from MT5 (copy_ticks_range) month-by-month and writes
#   tab-separated CSV(s) under data/ticks/. With --split-days it cuts each month
#   into fixed N-day CALENDAR windows named by their start day, deletes the full
#   month file, and sub-caps a hot window at 95 MiB so every part stays under
#   GitHub's 100 MiB push limit -- the AGREED archive format:
#       data/ticks/XAUUSD_TICK_YYYYMM_D<start>_pN_ELEV8.csv
#       (_D1_p1 = days 1-3, _D4_p1 = 4-6, ...; a window over 95 MiB adds _p2, _p3)
#   A tick ALWAYS lands in the same date window, so re-splits are deterministic
#   and grow-only (unlike a byte split, whose part boundaries move every re-sync).
#   Times are stored in CHART time = GMT+3 (the MT5 server clock), same as the M1
#   archive.
#
#   This is the archive the TICK-aware backtest consumes:
#     - tools/backtest_hybrid.py  (--ticks data/ticks/XAUUSD_TICK_*_ELEV8.csv) and
#       the CLI snapshots' 2026 backtest sections (5 & 6), and
#     - tools/tick_backtest.py and the tick sweeps.
#   (Consumers glob _*_ELEV8.csv and re-sort by timestamp, so the part naming
#   never matters to them -- but EVERY generator must emit the _D<start>_pN format
#   so the committed archive stays uniform; tests/test_tick_archive_format.py
#   fails CI on any legacy _pN / unsplit tick file.)
#
# WHY 95 (not 100):
#   GitHub HARD-REJECTS any file >= 100 MiB on push (and warns above 50 MiB).
#   The 95 MiB within-window sub-cap leaves margin so a push never bounces;
#   June's volatile 16-18 / 22-24 windows are why a window can need a _p2.
#
# IMPORTANT — broker tick window is SHORT:
#   MT5 only returns the ticks YOUR BROKER still stores. Tick history is far
#   shorter than M1 -- often just days to a few weeks (some brokers keep no
#   tick history at all). So set --start-date to the recent window you actually
#   want; requesting more just yields whatever is available. The committed
#   archive currently spans 2026-05-05 -> 2026-06-24 (May + June).
#
# IMPORTANT — --split-days is a PACKAGING step that removes the full month file.
#   On --merge it auto-reassembles the committed parts (both _D<start> date parts
#   and any legacy _pN size parts) into the full month, appends only newer ticks,
#   then re-splits by day. That reassemble round-trip is fine for the current
#   month; for a big month synced often, prefer the working-file TIP below.
#
# Dates are CHART-time GMT+3; --end-date is EXCLUSIVE.
#
# NOTE: the tick archive is refreshed by THIS snapshot (run it deliberately when
# the market is open). The strategy snapshots' backtest sections (5 & 6) run with
# --sync-ticks false -- they read the committed archive and do NOT auto-sync,
# because the in-backtest tick sync could collide with another process holding the
# parts (Windows WinError 32) and is unnecessary on a closed-market weekend.
# When --sync-ticks IS used, the refresh is an incremental APPEND (export_ticks
# --merge --split-days, default 3): it resumes from the last tick and re-splits
# ONLY the affected day window(s) -- 0 new ticks is a true no-op (no reassemble),
# so it is safe to leave on if you prefer. Run this standalone to (re)build the
# tick archive, or for a custom date range.
#
# WINDOWS FILE LOCKS (WinError 32):
#   Every tick-archive delete (the part/source unlink after a join or split) now
#   goes through robust_remove(), which RETRIES a Windows sharing violation
#   (WinError 32) with capped backoff (~16s) instead of aborting the sync. So a
#   freshly-written part momentarily held by Windows Defender, the Search indexer,
#   Explorer's preview pane, or a parallel run no longer leaves a half-reassembled
#   archive (both the joined ..._D<start>_<TAG>.csv AND its _pN parts -> duplicate
#   ticks). Belt-and-suspenders, so it never even waits: (1) EXCLUDE data\ticks
#   from your antivirus (Windows Security -> Virus & threat protection ->
#   Exclusions -> Add folder), and (2) prefer running THIS standalone sync over a
#   backtest's --sync-ticks true, with no other backtest/MT5/Explorer holding the
#   files. If robust_remove still gives up after ~16s, close whatever holds the
#   file and re-run -- the append is incremental, so a re-run is cheap.
#
# ===== RESYNC TICKS (run from the repo root) =====
python tools/export_ticks.py `
  --symbol XAUUSD `
  --start-date 2026-05-01 `
  --end-date 2026-06-26 `
  --output-dir data/ticks `
  --mt5-server-offset 3 `
  --split-days 3 `
  --progress

# ===== CONTINUE FROM THE LATEST STATE (incremental --merge) =====
# Add only the NEW ticks since the last fetch. --merge resumes from the last
# recorded tick and appends just the newer ones (so ticks already archived but
# since aged out of the broker's tick window are preserved -- --overwrite would
# lose them). Against the COMMITTED day-window archive, --merge reassembles the
# _D<start>_pN parts into the full working month, appends, and re-splits by day
# (deterministic: a tick always lands back in the same window). Set the dates to
# the month you are extending (same month = grow it forward; a new month = first
# fetch of it). Tip: just sync the CURRENT month -- past months don't gain ticks.
python tools/export_ticks.py `
  --symbol XAUUSD `
  --start-date 2026-06-01 `
  --end-date 2026-07-01 `
  --output-dir data/ticks `
  --mt5-server-offset 3 `
  --merge `
  --split-days 3 `
  --progress

# TIP -- avoid the reassemble round-trip on every update: keep the FULL working
# month file locally (do NOT commit it; it is >100 MiB) and --merge onto it WITHOUT
# any --split flag day-to-day; run --split-days 3 only right before you commit/push.
# Then `git add` just the _D<start>_pN parts. Full file = incremental working copy;
# date parts = what you commit.
